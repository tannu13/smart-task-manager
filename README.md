# Smart Task Manager with AI Briefing

## 1. Project Overview

Smart Task Manager with AI Briefing is a minimal task management backend built for a full-stack technical assignment. It exposes a REST API for creating, listing, and deleting tasks, stores data in PostgreSQL, and generates a short daily briefing for pending tasks using a real LLM integration.

When AI generation fails, the API falls back to a deterministic summary so the feature still returns a useful response instead of breaking the client flow.

Key capabilities:

- Task create, list, and delete APIs
- AI-generated daily briefing for pending tasks
- Fallback summary generation when the AI provider fails

## 4. Project Structure

This repository is organized as a small workspace, but the current submission only includes the API app.

- `apps/api` -> backend application
- `apps/api/src/routes` -> Express route definitions
- `apps/api/src/controllers` -> request/response handlers
- `apps/api/src/services` -> business logic and AI integration
- `apps/api/src/db` -> Drizzle connection and schema

## 5. Setup Instructions

The fastest way to run the project locally is:

### Prerequisites

- Node.js with Corepack enabled
- Docker and Docker Compose

### 1. Install dependencies

```bash
corepack enable
pnpm install
```

### 2. Configure environment variables

```bash
cp apps/api/.env.example apps/api/.env
cp apps/api/.env.docker.example apps/api/.env.docker
```

Update `apps/api/.env` with a valid `GOOGLE_GENERATIVE_AI_API_KEY`.

For the test environment, create matching test env files:

```bash
cp apps/api/.env.docker.example apps/api/.env.docker.test
cat > apps/api/.env.test <<'EOF'
PORT=4000
NODE_ENV=test
APP_STAGE=dev
DATABASE_URL=postgresql://myuser_test:mypassword_test@localhost:5433/smart_task_test
GOOGLE_GENERATIVE_AI_API_KEY=ASDF
EOF
```

Then update `apps/api/.env.docker.test` to use the test database values:

```env
PROJECT_NAME=smart_task_test
DB_PORT=5433
DB_NAME=smart_task_test
DB_USER=myuser_test
DB_PASSWORD=mypassword_test
```

### 3. Start PostgreSQL with Docker

```bash
cd apps/api
docker compose --env-file .env.docker up -d
```

This starts PostgreSQL on `localhost:5432`.

To start the separate test database:

```bash
cd apps/api
docker compose --env-file .env.docker.test up -d
```

This starts the test PostgreSQL instance on `localhost:5433`.

### 4. Run database migrations

From the repository root:

```bash
pnpm --dir apps/api db:migrate
```

### 5. Start the backend

```bash
pnpm --dir apps/api dev
```

The API starts on `http://localhost:4000`.

## 6. Environment Variables

The API reads environment variables from `apps/api/.env`.

| Variable                       | Required | Description                                                                      |
| ------------------------------ | -------- | -------------------------------------------------------------------------------- |
| `NODE_ENV`                     | Yes      | Runtime mode. Expected values: `development`, `production`, or `test`.           |
| `APP_STAGE`                    | Yes      | Application stage flag used by the env schema. Expected values: `dev` or `prod`. |
| `PORT`                         | Yes      | Port used by the Express server.                                                 |
| `CORS_ORIGIN`                  | Yes      | Comma-separated list of allowed frontend origins for CORS.                       |
| `DATABASE_URL`                 | Yes      | PostgreSQL connection string used by Drizzle and the API.                        |
| `GOOGLE_GENERATIVE_AI_API_KEY` | Yes      | API key for the Gemini provider used to generate task briefings.                 |

Example:

```env
NODE_ENV=development
APP_STAGE=dev
PORT=4000
CORS_ORIGIN=http://localhost:3000
DATABASE_URL=postgresql://myuser:mypassword@localhost:5432/smart_task
GOOGLE_GENERATIVE_AI_API_KEY=your_google_generative_ai_api_key
```

## 7. API Overview

Base URL:

```text
http://localhost:4000
```

### `POST /tasks`

Creates a new task.

Request:

```http
POST /tasks
Content-Type: application/json

{
  "title": "Finish backend tests"
}
```

Response:

```json
{
  "data": {
    "task": {
      "id": "2c747f7d-f1fe-4ea9-a910-b4c3dc8a9a54",
      "title": "Finish backend tests",
      "isCompleted": false,
      "createdAt": "2026-03-29T08:00:00.000Z"
    }
  }
}
```

### `GET /tasks`

Returns all tasks ordered by newest first.

Request:

```http
GET /tasks
```

Response:

```json
{
  "data": {
    "tasks": [
      {
        "id": "2c747f7d-f1fe-4ea9-a910-b4c3dc8a9a54",
        "title": "Finish backend tests",
        "isCompleted": false,
        "createdAt": "2026-03-29T08:00:00.000Z"
      }
    ]
  }
}
```

### `DELETE /tasks/:id`

Deletes a task by UUID.

Request:

```http
DELETE /tasks/2c747f7d-f1fe-4ea9-a910-b4c3dc8a9a54
```

Response:

```http
204 No Content
```

Validation error example:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "details": [
      {
        "field": "id",
        "message": "Invalid task ID format"
      }
    ]
  }
}
```

### `GET /tasks/summary`

Generates a short daily briefing for all pending tasks.

Request:

```http
GET /tasks/summary
```

Response:

```json
{
  "data": {
    "summary": "Start with Finish backend tests, then move to Review API errors. Focus on the highest-impact work first and keep the momentum going.",
    "pendingTasksCount": 2
  },
  "meta": {
    "source": "ai"
  }
}
```

Empty-state response:

```json
{
  "data": {
    "summary": "You have no pending tasks.",
    "pendingTasksCount": 0
  }
}
```

## 8. AI Integration

The summary endpoint collects all tasks where `isCompleted = false`, converts them into a short bullet list, and sends that list to Gemini through the Vercel AI SDK.

The prompt is intentionally constrained: it asks the model to identify priorities, suggest what to start with, keep the answer under five sentences, and stay actionable. If the provider call throws, the service falls back to a local summary builder so the endpoint still returns a usable briefing with `meta.source = "fallback"`.

## 9. Testing

Run the API test suite with:

```bash
pnpm --dir apps/api test
```

Current tests cover:

- `GET /health`
- `POST /tasks`
- `GET /tasks`
- `DELETE /tasks/:id`
- `GET /tasks/summary` empty-state behavior
- Summary fallback logic when AI generation fails

The AI path is mocked in the fallback-focused service tests by passing a fake `generateSummary` implementation into `getSummaryWithFallback`.

Note: the tests expect a separate PostgreSQL test database configured via `apps/api/.env.test` and `apps/api/.env.docker.test` on port `5433`.

### 1. Create the test env files

Make sure these two files exist:

- `apps/api/.env.docker.test`

Suggested contents:

### 2. Start the test database with Docker Compose


You can use either push or migrations.

Push schema directly:

Run migrations:



### 6. Stop the test database when finished

```bash

```
